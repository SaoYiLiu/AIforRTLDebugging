## Rationale
- **Icarus still warns on `aes128_key_expansion.sv` line 80** (`constant selects in always_* … all bits will be included`): the `expand_next_rkey` `always_comb` if/else chain on `key_exp_steps_ff[k]` makes Icarus merge every branch, so all `expanded_key_ff[…]` reads and all `next_rkey` writes can fire together — round keys K1–K10 are corrupted.
- **Harness evidence matches bad round keys, not a hang:** Run 0 fails at 240 ns with `0x20f710763f6d871fa87bee19396423fd != 0xebdfac91a5133de82760ac889f4a64d7` after key expansion + 11 decrypt rounds; wrong expanded keys yield wrong but complete 128-bit output.
- **Fix: remove conditional `always_comb` key-path logic** — precompute `nrk0..nrk9` with **continuous assigns** and dedicated sbox instances (constant slices only), then select the active slot with a one-hot **assign mux** for `expanded_key_nx`. This eliminates the line-80 warning class.
- **`aes128_decrypt.sv` xtimes chain (lines 143–151) is already correct** — each `×{02}` reduces on its own MSB; InvShiftRows, `(11-round_ff)` indexing, round 1 initial AddRoundKey, and round 11 no-MixColumns are unchanged.
- **Key-expansion XOR/RotWord order is retained** (`W[4g+0] ^ RconXor` then chain through `W[4g+1..3]`); only the Icarus-safe implementation form changes.

                             {xtimes04[i][j][NBW_BYTE-2:0], 1'b0};

            xtimes0b[i][j] = xtimes08[i][j] ^ xtimes02[i][j] ^ AddRoundKey[i][j];

        MixColumns[3][i] = xtimes0b[0][i] ^ xtimes0d[1][i] ^ xtimes09[2][i] ^ xtimes0e[3][i];
    end

                RoundOut[i][j] = current_data_ff[i][j] ^ expanded_key[NBW_EX_KEY-(11-round_ff)*NBW_KEY-(4*j+i)*NBW_BYTE-1-:NBW_BYTE];

                RoundOut[i][j] = AddRoundKey[i][j];
            end else begin
                RoundOut[i][j] = current_data_ff[i][j];
            end
        end
    end
