## Rationale
- **Line L008** implements `(A<<8)-(A<<2)+A`, which algebraically equals **A × 253** (256 − 4 + 1), not **A × 0xFB (251)**.
- The constant **0xFB = 251 = 256 − 5 = 256 − 4 − 1**, so the correct shift/add decomposition is `(A<<8) - (A<<2) - A`.
- At the first mismatch (**t = 15 ps**, **A = 0xC8 = 200**): `B_ref = 0xC418 = 50200 = 200×251`, while `B_dut = 0xC5A8 = 50600 = 200×253` — off by exactly **2×A**.
- At **A = 1** (t = 175 ps): `B_ref = 0xFB (251)` vs `B_dut = 0xFD (253)`, again a consistent **+2×A** error.
- The mismatch pattern (414/417 failures, only **A = 0** passes) matches a wrong multiplier constant affecting every nonzero input equally.
- **Fix:** change `+A` to `-A` on line L008 so the expression computes A × 0xFB with the same minimal shift/subtract structure.
