## Rationale
- **Suspected bug (L013):** The shift register rotates in the wrong direction: `q <= {q[5], q[4:0]}` is a **rotate right**, which feeds the MSB back into the MSB position. Because `q[5]` is `0` after reset (`6'b001011`), every rotation keeps `q[5]` at `0`, so `data <= q[5]` (L019) never becomes `1`.

- **Waveform evidence:** `data_dut` stays at `0` for the entire trace while `data_ref` begins toggling at t=45 ps with the expected `001011` pattern. The first mismatch is at t=50 ps (198/405 samples wrong), exactly when the reference expects the third bit (`1`) but the DUT still outputs `0`.

- **Causal trace:** Reset releases at t=15 ps with `q = 001011`. After three posedges (t=15, 25, 35), the reference outputs `0, 0, 1`; the DUT outputs `0, 0, 0` because `q[5]` never advances to `1` under right-rotation.

- **Correct behavior:** MSB-first serial output requires a **rotate left**: output `q[5]`, then shift `{q[4:0], q[5]}` so the next bit moves into the MSB position. This matches the reference model (`RefModule.sv` line 14).

- **Fix applied:** Change L013 from `{q[5], q[4:0]}` to `{q[4:0], q[5]}`. Reset logic (L010–L011, L016–L017) and the output tap on `q[5]` (L019) are unchanged.

- **Verification:** Simulation with the fix passes with 0 mismatches in 405 samples.

## FixedTopModule
```verilog
module TopModule(
    input clk,
    input rst_n,
    output reg data
    );
     
    reg [5:0] q;
     
    always@(posedge clk or negedge rst_n)
        if (!rst_n)
            q <= 6'b001011;
        else
            q <= {q[4:0], q[5]};
     
    always@(posedge clk or negedge rst_n)
        if (!rst_n)
            data <= 1'd0;
        else
            data <= q[5];
endmodule
```
