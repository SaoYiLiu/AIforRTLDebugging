- The CVDP harness already builds and runs the DUT with **Icarus Verilog** (`iverilog` + `vvp`) and cocotb, so a local iverilog run would use the same engine and would not expose anything the harness cannot already show.
- The failure is a **final functional mismatch** (`o_data` vs `model.data`) with both 128-bit hex values logged (`0x16b50503…` vs `0x82357fe9…`). That is enough to know the bug is in AES/key-schedule logic, not elaboration, parameters, or compile/import.
- The cocotb test drives **1000 random key/plaintext pairs** per reset (`runs = 1000`), but the harness excerpt does not record the specific failing `key`/`data` values. A standalone TB cannot faithfully reproduce the exact failing vector without `harness_library` and the same Python random sequence.
- Prior static analysis already points to concrete RTL issues (initial AddRoundKey source, parallel key-expansion chaining, round-key selection from `expanded_key_nx` vs registered state). Those are best fixed by aligning RTL with the reference model’s cycle semantics, not by inspecting a 1408-bit expanded-key waveform.
- The harness GPI warnings (`Unable to create cypher_logic`, `next_data`, etc.) suggest **limited internal visibility** through the existing flow; a VCD dump would mostly show ports plus hard-to-interpret wide internal buses, with low payoff for the next fix iteration compared to correcting the identified mux/expansion paths.
