## Rationale
- **Primary bug (L021):** The write path uses inverted enable logic — `!write_en ? write_data : myRAM[write_addr]` writes when `write_en` is **low** and holds when it is **high**, which is the opposite of the spec.
- **Waveform evidence at t=80 ps:** After reset, the testbench writes `4'b1010` to address 0 and `4'b0101` to address 1 with `write_en=1` (t=25–50). At t=75, `read_en=1` and `read_addr=1`; the reference reads `1010` (addr 0 was sampled on the prior cycle) while the DUT still outputs `0` because no writes occurred while `write_en` was asserted.
- **376/427 mismatches on `read_data`:** Nearly all read samples are wrong because RAM contents are never updated during normal write cycles; only spurious writes happen when `write_en` drops.
- **Read logic (L027) is correct:** `read_data <= read_en ? myRAM[read_addr] : read_data` matches the reference behavior (registered read, hold when disabled).
- **Fix applied:** Change L021 to `write_en ? write_data : myRAM[write_addr]` so writes occur when `write_en` is asserted. Verified: 0 mismatches against the ChipBench reference testbench.

## FixedTopModule
```verilog
module TopModule(
	input clk,
	input rst_n,
	
	input write_en,
	input [7:0]write_addr,
    input [3:0]write_data,
	
	input read_en,
	input [7:0]read_addr,
	output reg [3:0]read_data
);
    reg [3:0] myRAM [7:0];
    
    reg [8:0] i;
    always@(posedge clk or negedge rst_n) begin
        if(~rst_n)
            for(i=0;i<256;i=i+1)
                myRAM[i] <= 0;
        else
            myRAM[write_addr] <= write_en ? write_data : myRAM[write_addr];
    end
    always@(posedge clk or negedge rst_n) begin
        if(~rst_n)
            read_data <= 0;
        else
            read_data <= read_en ? myRAM[read_addr] : read_data;
    end
endmodule
```
