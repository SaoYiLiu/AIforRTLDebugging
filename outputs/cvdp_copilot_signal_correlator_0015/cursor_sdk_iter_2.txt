## Rationale
- **Harness evidence:** `test_clamping_logic` fails with `Expected 15, got 0` at 21 ns when both inputs are `0b11111111` — the output never leaves reset value.
- **Lines 16–21:** The `else` branch computes `sum` but never assigns it to `correlation_output`; only the reset path (line 14) updates the output, so it stays at `0`.
- **Line 19:** Each matching bit adds `1` via `(input_signal[i] & reference_signal[i])`, but the spec requires **+2** per matching bit (8 matches → raw sum `16`, not `8`).
- **Lines 10, 19:** `sum` is only 4 bits wide; the maximum weighted sum is `8 × 2 = 16`, which needs at least 5 bits before clamping.
- **Missing clamping:** When the raw sum exceeds `15`, `correlation_output` must saturate to `15` (the clamping test case: all bits match → sum `16` → output `15`).
- **Reset behavior (lines 13–14):** Reset correctly drives `correlation_output <= 0`; the fix preserves that and adds the missing computation/clamp in the non-reset path.

## PatchedFiles
### rtl/signal_correlator.sv
```verilog
module signal_correlator(
    input clk,
    input reset,
    input [7:0] input_signal,
    input [7:0] reference_signal,
    output reg [3:0] correlation_output // 4-bit output
);

integer i;
reg [4:0] sum;

always @(posedge clk or posedge reset) begin
    if (reset) begin
        correlation_output <= 0;
    end else begin
        sum = 0;
        for (i = 0; i < 8; i = i + 1) begin
            if (input_signal[i] & reference_signal[i])
                sum = sum + 2;
        end
        if (sum > 15)
            correlation_output <= 15;
        else
            correlation_output <= sum[3:0];
    end
end

endmodule
```
